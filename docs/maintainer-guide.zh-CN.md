# GitHub 新手维护手册

这份手册只讲日常需要做什么。

## 先理解四个词

- **Issue**：问题单或需求单，像待办事项。
- **Branch**：独立修改区，不直接动正式版本。
- **Pull Request（PR）**：请求把 Branch 的修改合入正式版本。
- **Release**：对外发布的稳定版本。

## 公开仓库是否人人都能改？

不能。

其他人可以阅读、复制、Fork、提 Issue 和 PR，但不能直接改你的 `main`。只有你和你邀请的协作者有写入权限。不要因为对方提过一次好建议就授予协作者权限。

## 每周看一次

打开仓库后看四处：

1. **Actions**：是否全绿；
2. **Issues**：有没有真实问题或需求；
3. **Pull requests**：有没有别人提交修改；
4. **Security**：有没有 Dependabot 或密钥警报。

没有异常，不需要为了活跃而乱提交。

## 收到 Issue 怎么处理？

先问三件事：

1. 是什么业务问题？
2. 能否用合成数据复现？
3. 是否属于通用需求，而不是某一家企业的私有规则？

然后添加标签：

- `bug`：现有功能结果错误；
- `enhancement`：新功能；
- `documentation`：文档问题；
- `question`：使用问题；
- `security`：不要公开讨论，转私下处理。

## 收到 PR 怎么处理？

不要只看代码能否运行。按顺序检查：

1. 有没有真实商户数据、接口或凭据；
2. 指标口径是否写清；
3. 测试是否覆盖变化；
4. CI 是否通过；
5. 是否会改变已有结果；
6. 文档是否更新；
7. 修改是否足够小，能独立理解。

不清楚就要求修改，不必急着合并。

## 自己修改仓库的推荐流程

```powershell
cd E:\Codex\open_source\restaurant-ops-toolkit
git switch main
git pull
git switch -c codex/简短功能名
```

修改完成后：

```powershell
python scripts/prepublish_scan.py
python -m unittest discover -s tests -v
git add 具体文件
git commit -m "简短说明"
git push -u origin 当前分支名
```

然后在 GitHub 创建 PR，等待 CI 通过，再合并。尽量不要直接向 `main` 推送。

## 什么时候发版本？

- 修复错误但不增加新功能：`v0.1.1`
- 增加兼容的新功能：`v0.2.0`
- 出现不兼容变化：Alpha 阶段也必须明确写迁移说明

发布前：

1. 所有测试通过；
2. 安全扫描通过；
3. 更新 `CHANGELOG.md`；
4. 确认 README 示例仍能运行；
5. 确认没有真实业务材料进入 Git 历史。

## 什么指标说明仓库有价值？

不要只看 Star。更重要的是：

- 有多少真实问题被解决；
- 有多少人成功运行示例；
- Issue 是否能得到回应；
- PR 是否被认真审核；
- 版本是否持续发布；
- 计算错误和人工整理时间是否下降；
- 是否出现外部贡献者或实际使用案例。

## 遇到恶意或混乱内容

你可以关闭 Issue、锁定讨论、隐藏评论或限制互动。公开不代表必须接受所有要求，也不代表你有义务立即回复。
