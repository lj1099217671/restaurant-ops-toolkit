# 为什么下一步是“Excel 到经营预警一页纸”

## 判断依据

这个优先级不是因为 Excel 技术简单，而是因为它同时满足四个条件：

1. **真实入口**：多数餐饮从业者能获得 Excel，却拿不到稳定 API；
2. **建立信任**：先暴露数据问题，再给经营结论；
3. **体现价值**：用户能从原始表直接获得可读的一页纸；
4. **方便维护**：输入、校验、计算和报告可以分层测试。

## 为什么不先只做漂亮看板？

如果底层表存在重复行、缺日期、门店重名或渠道差额，漂亮看板只会更快传播错误。

## 为什么不一次加入全部评价能力？

营业数据主要是确定性计算；评价判断涉及语义、上下文和人工复核。两者应共用数据质量和报告框架，但不应为了赶版本而混成一个不可验证的大功能。

合理顺序是：

1. Excel 数据质量闸门；
2. 营业预警一页纸；
3. 门店生命周期和同店分析；
4. 评价预警及人工复核；
5. 营业与评价合并的一页纸。

## 对仓库维护的帮助

这条路线会形成清晰模块：

- `import`：读取 Excel；
- `quality`：生成质量问题和通过状态；
- `metrics`：确定性指标；
- `alerts`：规则化预警；
- `reports`：一页纸呈现。

每个模块都能独立测试，外部贡献者也能明确选择任务，不会把所有业务逻辑堆在一个脚本中。
