# 项目治理 | Governance

## 谁拥有维护权？

本仓库当前由 GitHub 账号 `lj1099217671` 所有。

公开仓库允许所有人：

- 阅读和下载代码；
- Fork 到自己的账号；
- 创建 Issue、Discussion 或 Pull Request。

公开仓库不会让所有人自动获得写入权。只有：

- 仓库所有者；
- 所有者明确邀请的协作者；

才能直接写入仓库。个人账号仓库中的协作者拥有写入能力，因此只应邀请长期、可信且确实需要直接维护的人。

## 角色

### 使用者 | Users

使用项目、报告问题、提出需求，不具有写入权限。

### 贡献者 | Contributors

通过 Issue、文档或 Pull Request 做出贡献，不具有自动合并权限。

### 协作者 | Collaborators

由所有者邀请，可以维护 Issue、审查或提交代码。授予前必须确认其职责和安全意识。

### 所有者 | Owner

负责仓库设置、权限、发布、安全响应和最终合并决定。

## 决策原则

1. **业务口径优先于代码便利**：先定义指标，再实现代码。
2. **可复核优先于智能推断**：核心数字必须由确定性代码计算。
3. **公开通用方法，保护企业资产**：不以开源名义暴露商户数据和私有系统。
4. **小步发布**：一个 PR 解决一个明确问题。
5. **保留人工判断**：模糊门店映射、评价分类和异常数据不得被强制自动化。

## 合并条件

进入 `main` 前应满足：

- CI 和安全扫描通过；
- 至少一名维护者完成审查；
- 指标变化有文档和测试；
- 不包含真实商户数据或凭据；
- 重大兼容性变化有迁移说明。

当前只有一名所有者时，所有者可以完成自审，但应优先通过 Pull Request 留下公开审查记录。

## 何时考虑 GitHub Organization？

当出现两名以上长期维护者、多个关联仓库或需要更细权限时，应将项目转入 GitHub Organization，并使用 Read、Triage、Write、Maintain、Admin 等细粒度角色。
